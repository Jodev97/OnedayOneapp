# 🚀 One App a Day -  de Desarrollo Moderno

> **Construyendo portafolio técnico sólido, una app a la vez.**
> 
> Un reto  enfocado en **técnicas de desarrollo moderno**: SDD,BDD, TDD, Clean Code, Design Patterns, Testing, Arquitectura escalable y **integración de AI Agents**.

---

## ¿Qué es One App a Day?

**One App a Day** es un reto **permanente y flexible** donde desarrolladores de todos los niveles construimos una aplicación con un enfoque en **técnicas de desarrollo de calidad**, no en cantidad.

### 🎯 Objetivos

✅ **Practicar técnicas modernas** - SDD,BDD, TDD, Clean Code, SOLID, Design Patterns  
✅ **Compilar portafolio técnico** - Código de calidad, no solo cantidad  
✅ **Integración de IA** - Aprender a trabajar con AI Agents (Claude, OpenAI, etc)  
✅ **Comunidad de aprendizaje** - Feedback, code reviews, conocimiento compartido  
✅ **Permanente** - Sin deadline, a tu ritmo, sin presión  

### 💡 El Diferenciador

No es "30 apps en 30 días". Es sobre hacer apps **BIEN**.

Cada aplicación demuestra:
- ✅ Código limpio y tipado
- ✅ Tests (unitarios, integración, E2E)
- ✅ Documentación clara
- ✅ Decisiones arquitectónicas thoughtful
- ✅ Uso de herramientas modernas
- ✅ Buenas prácticas aplicadas

## 🎯 Reglas del Reto

### ✔️ Requisitos Mínimos

Una app cuenta como **completada** si cumple:

1. **Código funcional y limpio**
   - ✅ Código que ejecuta sin errores
   - ✅ Siguiendo estándares del lenguaje
   - ✅ Legible y bien estructurado

2. **Técnica moderna aplicada** (elige al menos 1)

   - ✅ TDD (tests primero)
   - ✅ TypeScript strict mode
   - ✅ Clean Code principles
   - ✅ SOLID principles
   - ✅ Design Pattern específico
   - ✅ Arquitectura escalable
   - ✅ Testing exhaustivo
   - ✅ CI/CD implementado
   - ✅ AI Agent integration

3. **Documentación clara**
   - ✅ README explicando qué hace
   - ✅ Arquitectura documentada
   - ✅ Setup instructions
   - ✅ Aprendizajes compartidos

4. **Código público**
   - ✅ GitHub repo
   - ✅ Historia de desarrollo visible

### ❌ No Cuentan

- ❌ Copiar tutoriales sin adaptación
- ❌ Código sin estructura
- ❌ Sin documentación
- ❌ Abandonado a mitad
- ❌ Sin commits (o 1 commit gigante)

## 🚀 Cómo Participar

### Opción 1: Mejorar una app existente

```bash
# Clonar el repo
git clone https://github.com/Jodev97/OnedayOneapp.git
cd OnedayOneapp

# Entrar a una app
cd 01-simple-ecommerce

# Crear rama
git checkout -b feature/tu-mejora

# Hacer cambios
# ...

# Commit y push
git add .
git commit -m "feat: descripción de tu mejora"
git push origin feature/tu-mejora

# Crear Pull Request en GitHub
```

### Opción 2: Crear una app nueva

1. **Fork este repo** (si quieres contribuir)
2. **Crea una carpeta** con nombre `0X-nombre-app`
3. **Desarrolla tu app** siguiendo las reglas del reto
4. **Documenta en README**
   - Qué hace
   - Tech stack
   - Cómo ejecutar
   - Aprendizajes
5. **Commits claros** mostrando progreso
6. **Crea un Pull Request** (si colaboras) o tu propio repo

### Opción 3: Tu propio repo

Si prefieres mantener tu propio repositorio:

1. Crea tu repo `OneAppADay` (o nombre similar)
2. Haz una app siguiendo las reglas
3. **Comparte el link en:**
   - Comentarios de ForoBeta
   - Twitter/X con #OneAppADay
   - LinkedIn
   - Discord/Slack comunitarios

## 💡 Aprendizajes

### Insights Generales

- **Especificaciones claras = Código mejor:** Cuando usas Spec-Driven Development con AI Agents, el contexto es clave
- **TypeScript strict mejora calidad:** Sin "any" implícitos, el código es más robusto
- **Tests desde el inicio:** E2E tests con Playwright son más amigables que muchos creen
- **Arquitectura escalable:** Desde el día 1, piensa en cómo pasaría a producción

Ver carpeta individual de cada app para aprendizajes específicos.

## ❓ FAQ

**P: ¿Cuánto tiempo debo dedicar por app?**
R: Depende de la complejidad. Generalmente 8-16 horas para una app "buena" (código + tests + docs).

**P: ¿Tengo que hacer una app CADA semana?**
R: No. Es permanente. Haz una cuando tengas tiempo y energía.

**P: ¿Puedo usar mis propios frameworks?**
R: Absolutamente. El foco está en la técnica, no en la tech stack.

**P: ¿Cómo contribuyo?**
R: Fork del repo, crea rama, haz cambios, pull request. Ver [CONTRIBUTING.md](./CONTRIBUTING.md).

**P: ¿Puedo hacer una app de X tecnología?**
R: Sí. Cualquier lenguaje/framework. Pero asegúrate de documentar bien.

**P: ¿Hay premio?**
R: Hall of Fame, visibilidad comunitaria, y un portafolio técnico increíble.
